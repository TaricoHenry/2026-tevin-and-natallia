# Wedding RSVP Admin + Public API

This project is a Firebase-backed RSVP system for managing wedding guest households and collecting responses.

It includes:

- a public RSVP API for guests
- a protected admin API for household management
- a React admin dashboard
- a Firestore bulk import script for initial guest loading

## What changed in this version

This version applies the requested system-managed field rules:

- the 6-character household code is generated by the backend and is not user-enterable
- the unique URL is generated from the household code and is not user-enterable
- each `memberId` is generated by the system and is not user-editable
- editing a household preserves the existing household code and URL
- editing a household preserves existing member IDs and only generates IDs for newly added members

## Architecture overview

### Backend

The backend uses:

- `express` for routing
- `firebase-functions/v2/https` for deployment as an HTTP function
- `firebase-admin` and Firestore for persistence
- `joi` for request validation
- `cookie-parser`, `jsonwebtoken`, and `bcryptjs` for admin authentication
- `cors` for browser access control

### Frontend

The admin dashboard is a React component that:

- authenticates against `/v1/admin/login`
- loads household data from `/v1/admin/households`
- creates households without exposing system-generated identifiers
- edits household/member details while keeping system identifiers read-only
- exports dashboard data as CSV

### Database model

Firestore collection:

- `InvitedGuests`

Each document represents one household.

Example document:

```json
{
  "code": "AB12CD",
  "uniqueUrl": "https://tevinandnatalie.com/AB12CD",
  "household": "The Smith Family",
  "householdSize": 3,
  "allResponded": false,
  "respondedAt": null,
  "members": [
    {
      "memberId": "AB12CD_1",
      "name": "John Smith",
      "personalizedAddy": "Mr. John",
      "rsvp": null,
      "respondedAt": null
    },
    {
      "memberId": "AB12CD_2",
      "name": "Jane Smith",
      "personalizedAddy": "Mrs. Jane",
      "rsvp": "yes",
      "respondedAt": "2026-04-02T12:00:00.000Z"
    }
  ],
  "createdAt": "Firestore server timestamp",
  "updatedAt": "Firestore server timestamp"
}
```

## System-managed fields

These values are controlled by the backend:

### Household code

- always 6 characters
- uppercase alphanumeric
- generated automatically
- immutable after creation
- used as the Firestore document ID

### Unique URL

- always derived from the household code
- format: `https://tevinandnatalie.com/{CODE}`
- immutable after creation

### Member ID

- generated automatically by the backend
- format: `{CODE}_{index}`
- existing IDs are preserved on update
- new members receive newly generated IDs

## Public API behavior

Guests use their household token/code to:

- check whether their invite token exists and is still usable
- load their household details
- submit RSVP responses per member

A household is considered fully responded only when every member has RSVP value `yes` or `no`.

## Admin API behavior

Admins can:

- log in and receive a session cookie
- log out
- list all households
- create households
- update household/member content

Admins cannot directly set:

- `code`
- `uniqueUrl`
- `memberId`

Those are backend-owned fields.

## Local development notes

### Environment variables

These should be supplied securely in production:

- `ADMIN_USERNAME`
- `ADMIN_PASSWORD_HASH`
- `ADMIN_JWT_SECRET`

Current code has development fallbacks, but production should not rely on them.

### CORS origins

Allowed origins currently include localhost and the production sites defined in `index.js`.

### Cookie behavior

Admin auth uses an HTTP-only cookie named `__session`.

Important details:

- `httpOnly: true`
- `secure: true`
- `sameSite: "lax"`
- `path: "/"`
- 8 hour expiration

For local non-HTTPS testing, `secure: true` can block cookie persistence unless you proxy through HTTPS or adjust local behavior.

## Running the import script

The import script loads CSV data into Firestore as household documents.

### Expected CSV headers

```text
name, personalizedAddress, personalizedHousholdName, housholdSize, rsvp, Unique URL
```

The script tolerates the existing spelling variants from the sheet.

### Usage

```bash
node importGuests.js ./csv/guestList.csv
```

### Import behavior

The script:

1. reads the CSV
2. groups rows into households
3. reuses an existing code from `Unique URL` when present
4. otherwise generates a unique 6-character code
5. builds one Firestore document per household
6. generates `memberId` values for each member
7. calculates `allResponded`

## Validation rules

### Public RSVP request

- requires `responses`
- each response must include:
  - `memberId`
  - `rsvp` of `yes` or `no`

### Admin household request

- requires `household`
- requires `householdSize`
- requires at least one member
- each member requires:
  - `name`
- optional:
  - `personalizedAddy`
  - `rsvp`
  - `memberId` in update payloads only for preserving existing identities

Unknown fields are stripped by Joi validation.

## Admin dashboard UX

The dashboard now reflects the system-managed policy:

- when creating a household, code and URL show as generated placeholders
- when editing a household, code and URL are displayed read-only
- member IDs are shown read-only for existing members
- newly added members show `Generated on save`
- users can still edit household name, size, personalized addresses, member names, and RSVP values

## Deployment notes

The backend exports a Firebase HTTPS function named `api`.

```js
export const api = onRequest({ maxInstances: 3, concurrency: 20 }, app);
```

Typical deployment flow:

1. install dependencies
2. configure environment secrets
3. deploy Firebase Functions
4. deploy the frontend/admin client
5. verify CORS and cookie behavior in production
